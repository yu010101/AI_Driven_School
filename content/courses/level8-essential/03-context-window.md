## Context Window — AIの記憶容量を理解する

Context Window = AIの**短期記憶**。一度に覚えられる量に限りがある。超えると古い情報から忘れていく。

:::before-after{before="なぜか急にAIが忘れる" after="トークン管理で安定動作"}

公式認定でContext管理は必出テーマ。トークンの概念と最適化手法を押さえよう。

### トークンとは何か

トークン = AIがテキストを処理する最小単位。英語では1単語 ≒ 1〜1.5トークン、日本語では1文字 ≒ 1〜3トークン。日本語は英語の約2〜3倍のトークンを消費するため、Context管理がより重要になります。

### Context Windowのサイズ

| モデル | Context Window | 実用可能量（目安） |
|---|---|---|
| Claude Sonnet 4 | 200K トークン | 約160K（システム分を除く） |
| Claude Opus 4 | 200K トークン | 約160K（システム分を除く） |

### トークン配分の内訳（実例）

```
┌─────────────────────────────────────────────┐
│            200,000 トークン（上限）            │
├─────────────────────────────────────────────┤
│ システムプロンプト     :   5,000 (2.5%)       │
│ CLAUDE.md             :   2,000 (1.0%)       │
│ settings.json / 設定   :   1,000 (0.5%)       │
│ ── ここまで固定コスト ──                      │
│                                              │
│ 会話履歴（あなたの入力＋AIの応答）            │
│                       : 120,000 (60.0%)      │
│                                              │
│ ファイル読み込み内容   :  15,000 (7.5%)       │
│                                              │
│ 検索結果・ツール出力   :   7,000 (3.5%)       │
│                                              │
│ 残り（余裕バッファ）   :  50,000 (25.0%)      │
└─────────────────────────────────────────────┘
```

固定コスト（システム + CLAUDE.md + 設定）で約8,000トークンが常に消費されています。残りの192,000トークンが実際に使える量です。

### コンテキストが溢れる兆候

以下のサインが出たら、Context Windowが限界に近づいています：

1. **Claudeが直前の指示を忘れる** — 「さっき言ったこと」が消えている
2. **ファイルの内容を間違える** — 古い版のコードを参照している
3. **同じ質問を繰り返す** — 既に回答済みの情報を再度尋ねてくる
4. **応答が急に短くなる** — トークン節約のため出力が圧縮される
5. **自動で /compact が実行される** — Claude Codeが自動的に会話を圧縮する

### /compact の仕組み

`/compact` は会話履歴を**要約して圧縮**するコマンドです。

```
あなた: /compact

Claude Code: 会話を要約しています...
  変更前: 145,000 トークン
  変更後:  28,000 トークン
  節約量: 117,000 トークン (81% 削減)
```

**何が起きるか：**
- 会話の要点（変更したファイル、決定事項、現在のタスク）が保持される
- 個々のやりとりの詳細は失われる
- CLAUDE.mdやシステムプロンプトはそのまま残る

**いつ使うか：**
- 長い会話の途中で「そろそろ重くなってきた」と感じたとき
- 新しいタスクに移る前のリセットとして
- Claude Codeの応答速度が遅くなったとき

### 実務での使い方

**大規模リファクタリング時**
ファイルを1つずつ処理し、区切りごとに `/compact` を実行。一度に全ファイルを読み込まない。

```
# 悪い例：一度に大量のファイルを読み込む
claude "src/ 以下の全ファイルをリファクタリングして"

# 良い例：段階的に処理する
claude "src/components/Header.tsx をリファクタリングして"
# → 完了後
/compact
claude "次は src/components/Footer.tsx をリファクタリングして"
```

**CLAUDE.md の最適化**
CLAUDE.mdが長すぎると毎回のトークン消費が増える。必要最低限の指示に絞る。

```markdown
# 悪い例（500行のCLAUDE.md → 約3,000トークン消費）
このプロジェクトは2023年に開始され、React 18を使用しています。
チームメンバーは5人で、毎週水曜にスプリントレビュー...
（延々と続く背景説明）

# 良い例（50行のCLAUDE.md → 約300トークン消費）
- TypeScript strict mode
- テストは vitest を使用
- コミットメッセージは Conventional Commits
- CSS は Tailwind v4
```

**ファイル読み込みの工夫**
大きなファイルは全体ではなく必要な部分だけ読む。

### よくある失敗

**1. CLAUDE.mdに不要な情報を詰め込む**
プロジェクトの歴史や設計思想を長々と書く。Claudeが毎ターン消費するトークンが増え、会話に使える量が減る。

**2. /clear と /compact の違いを理解していない**
`/clear` は会話履歴を**完全削除**。`/compact` は**要約して圧縮**。進行中のタスクがある場合は `/compact` を使うこと。`/clear` すると全てのコンテキストが失われる。

**3. 巨大なログ出力をそのまま貼り付ける**
エラーログの全量（数千行）を貼り付けると一瞬でContextが埋まる。関連部分だけ抜粋する。

**4. 複数タスクを1セッションで処理しすぎる**
タスクAの文脈がタスクBに持ち越され、混乱の原因に。タスクが変わったら `/compact` または新セッション開始。

### コストへの影響

Context Windowの使い方はコストに直結します。入力トークン（Claudeに送る量）が多いほど料金が上がる。`/compact` で80%圧縮すれば、その分のAPIコストも削減できます。詳しくは次のレッスン「コスト管理」で解説します。

## Context Windowの使用量を確認しよう

効率的なContext管理のコツ：

- `/compact` で会話を要約・圧縮（進行中タスクを保持）
- `/clear` で会話履歴を完全リセット（新タスク開始時）
- 大きなファイルは**部分読み込み**で節約
- CLAUDE.mdの指示は**50行以内**を目標に簡潔に書く
- 日本語は英語の2〜3倍トークンを消費することを意識する

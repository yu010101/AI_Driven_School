# サイト全体設計

## 基本構造：二階建て設計

### 1F：公式サイト（成約装置）

**目的**: 信頼・商品導線・リード獲得

**KPI**: LINE/メール登録、無料教材DL、相談申込、教材購入、コミュニティ参加

**必須ページ**:
- `Home`: 一言で何者か／誰向けか／何が得られるか
- `About`: 背景＋発信実績＋価値観（AI×マーケ×非エンジニア開発）
- `Free教材`:
  - バイブコーディングの教科書（登録）
  - 0円マーケティングの教科書（登録）
- `Products/Services`: 今売るものが未確定でもOK（「予定」「ベータ」でも導線は作る）
- `Articles`: noteの"要約＋追記"を置く（canonical設計は後述）
- `Contact`: 相談/取材/案件

**勝ち筋**: 「無料教材」ページを2枚の強LPとして作り、X/noteの全導線をそこに集約

### 2F：SEO用DBサイト（集客装置）

**目的**: ロングテールを面で取りに行く（プログラマティックSEO/辞書型/比較/テンプレ配布）

**KPI**: 自然検索セッション、インデックス数、順位、CVR（登録率）

**構造**: `/knowledge/` 配下に3つのDBを配置

## 3つのDB設計

### DB① バイブコーディングDB（作る力）

**役割**: 
- 非エンジニア／初心者の「作れない不安」を潰す
- バイブコーディングの教科書への最短導線

**カテゴリ設計**:
```
/knowledge/vibe-coding/
  /what-is/        # 基礎概念
  /prompts/        # プロンプト集
  /mistakes/       # 失敗例
  /flows/          # フロー・手順
  /checklists/     # チェックリスト
```

**検索KW例**:
- バイブコーディング とは
- バイブコーディング プロンプト
- AI コーディング 失敗例
- 非エンジニア アプリ開発 方法

### DB② 個人開発・実装DB（作れる → 動く）

**役割**:
- 「作り方は分かった。でも実装で詰まる」を解消
- "できる人"感を一気に上げるゾーン
- 後の有料教材・講座の核

**カテゴリ設計**:
```
/knowledge/build/
  /features/      # 機能別（ログイン、決済、メール等）
  /stacks/        # 技術構成（Next.js+Supabase等）
  /templates/     # 雛形
  /troubles/      # エラー・詰まり
```

**例**:
- `/features/auth` - ログイン機能の作り方
- `/features/payment` - 決済機能の実装
- `/stacks/nextjs-supabase` - Next.js+Supabase構成
- `/templates/saas-admin` - SaaS管理画面テンプレ

### DB③ 0円マーケティングDB（作った → 売れる）

**役割**:
- 収益欲求に直撃
- 最もCVしやすい
- 「0円マーケティングの教科書」の母数を最大化

**カテゴリ設計**:
```
/knowledge/marketing/
  /tactics/       # 施策（Xスレッド、SEO等）
  /platforms/     # プラットフォーム別（X / note / SEO）
  /templates/     # コピペ用テンプレ
  /case/          # 事例
```

**例**:
- `/tactics/x-thread` - Xスレッド集客
- `/tactics/programmatic-seo` - プログラマティックSEO
- `/platforms/note` - note集客方法
- `/templates/sales-copy` - セールスコピー

## 3DBを分断させない設計（最重要）

### 内部リンクの思想

すべての記事に「次の一手」を用意:
- **作り方 → 実装**: バイブコーディング記事末尾 →「次は ログイン機能 を作ろう」
- **実装 → 売り方**: 実装記事末尾 →「作ったら Xで0円集客 しよう」
- **売り方 → 作り直し**: マーケ記事末尾 →「次は 改善のために作り直す」

**Google的には**: 「単なる辞書」ではなく「体系化された専門メディア」になる

### 公式サイトとDBの役割分担

| 項目 | 公式 | DB |
|------|------|-----|
| 信頼 | ◎ | △ |
| SEO | △ | ◎ |
| 成約 | ◎ | ○ |
| 更新頻度 | 低 | 高 |

**→ DBで集めて、公式で刈り取る**

## リードマグネット2本の最適配置

### 完全に役割分離

- **作る系DB → バイブコーディングの教科書**
- **売る系DB → 0円マーケティングの教科書**

### やってはいけない

- ❌ 1ページに2つ同時に置く
- ❌ どっちも同じ文言でCTA

## noteとの共存（SEOの食い合いを避ける）

**推奨**: noteは認知・拡散、サイトは完成版（追記/事例/テンプレ付き）

- note → 公式サイトへ送客
- サイト側がSEOで勝ちやすい

**理由**: noteにcanonicalを自由に入れられないので、実務上はこの方法が無難
